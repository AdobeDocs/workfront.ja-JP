# 書き込み品質基準

`writing-quality` スキルの詳細なルール。 SKILL.md ファイルは、最も一般的なルールをカバーしています。このファイルは、句読点、用語、大文字/小文字の例、プロシージャの書式設定に関する完全なリファレンスを提供します。

---

## 句読点

### 期間

- アイテムが文章を完成する場合は、リスト内でピリオドを使用します（「X メニューから、次の操作を実行できます」）。
- 項目が完全な文である場合は、リスト内のピリオドを使用します
- 期間はメインコンテンツでフォーマットしないでください（正解：「**完了**」をクリックします）。 /不正確：「**完了」をクリックしてください。**）
- ファイル拡張子（.pdf）でピリオドを使用

### コンマ

- オックスフォード・コンマは常に“動機、手段、機会”
- 「文書を開くには、開くをクリックしてください」という入門的なフレーズの後のカンマ。
- 日付の後の年のコマ中文：&quot;2016年1月2日、問題の…&quot;
- 手順の「then」の前にカンマを入力します。「**設定**」をクリックし、「**インターフェイス**」をクリックします。
- 月別コンマなし：「2016年1月」

### アポストロフィ

- 契約で使用する：しないでください、彼らは、それは非常に簡単です
- Singular possessive:&#39;s （Jen&#39;s computer）を追加する
- sで終わる複数の所有物：「（The Jonesesの）コンピューターのみを追加する
- 「it&#39;s」（it is）と「its」（it isに属する）を区別する

### コロン

- 直引用、固有名詞、箇条書きの後に続かない限り、コロンの後に小文字を使用します
- リストを紹介する文章/フラグメントの末尾で使用
- プロシージャ見出しの後にコロンがありません
- ビジュアル要素やコードセグメントを導入するコロンはありません

### セミコロン

- 連結せずに2つの句を連結する
- 句読点を含む個別のフレーズ
- 手順で太字のテキストに続いてセミコロンを太字にしないでください
- ハイパーリンクに含めない

### 引用符

- 「引用符」ではなく「引用符」と呼びます。
- 閉じる引用符は、コンマとピリオドの外に配置されます
- 疑問符の配置は、引用符または文が質問であるかどうかによって異なります

### ハイフン

- 可能な場合は、ハイフンを必要とする構造を避けます
- 名詞の前に複合修飾子をハイフネート：「読み取り専用メモリ」
- 1つの修飾子が特定の場合はハイフネーションを適用します。「copy-protected disk」
- 1つの修飾子が「12 ポイントフォント」である場合のハイフネーション
- キーの組み合わせにハイフンがありません – + （Ctrl+I）を使用します

### EM ダッシュ

控えめに使う。 代わりに文章を再構成することを検討してください。

### エン・ダッシュ

- 範囲：2005 ～ 2013
- マイナス記号または負の数
- 複合形容詞を別の単語に接続する：「ダイアログボックスの種類オプション」

### 楕円形

- 省略されたコードまたは構文
- ダイアログボックスが表示されることを示すUI
- 引用符で囲んだテキスト（文末尾を置き換える場合はピリオドを追加）

### スラッシュ

- 選択肢にスラッシュを使用しない – 「or」を使用します（「he/she」ではなく「he/she」）
- 組み合わせに使用：オン/オフスイッチ
- インターネット アドレスと分数に使用

## 用語

### UI要素

| 要素 | 正しい用語 |
|---------|-------------|
| 折りたたみ可能なオプションのリスト | メニュー（必要に応じて「ドロップダウン」を追加すると分かりやすくなります） |
| 使用できないコマンド | 「利用不可」（暗い、無効、グレー表示は使用不可） |
| フォームフィールドでポップアップ | ダイアログボックス |
| 名称未設定のボタン | ボタンのツールチップ名を使用する |
| エラーメッセージ/トースト通知 | 「メッセージ」（「メッセージが表示されます…」） |
| テキスト入力 | フィールド |

### 役割

| コンテキスト | 用語 |
|---------|------|
| Workfrontにおける役割 | システム管理者 |
| Workfrontを管理する人 | Workfront 管理者 |

### 上書きと上書き

これらは意味の異なる複数の単語です。コンテキストに適した単語を使用してください。

### ファイル拡張子

必ず、頭字語ではなくファイル拡張子を使用してください：「.pdf ファイルのアップロード」 （「PDFのアップロード」ではありません）。

## プロシージャのフォーマット

### 概要

必要な場合にのみ含めます。 推奨される形式（順序）:

1. 不定詞+コロン：「システム内のユーザーの一時パスワードを作成するには：」
2. 文章+期間：「システム内のユーザーの一時的なパスワードを作成する必要があります。」
3. Infinitive phrase + &quot;次の手順に従う&quot; + colon: &quot;ユーザーの一時パスワードを作成するには、次の手順に従います。&quot;

### 手順の規則

- シングルステップでも、すべての手順に番号付きリストを使用する
- クリック可能なアクションとUI要素をステップで太字にする
- 特定のアクション動詞を使用する：「type」、「select」、「click」 – 「set」または「specify」
- システム応答を記述する：発生する前または発生直後

システム応答の配置の例：

```
1. Click **Sharing**.
2. In the **Task Access** dialog box, enter the name of the user you want to give task access to.
```

### 条件付き手順とオプション手順

```
3. (Conditional) If you are a member of more than one team, select...
3. (Optional) To add an emoji, click...
3. (Conditional and optional) If you want to add a note...
```

### サブステップとフィールドリスト

必須コマンド（設定、指定、実行）を使用して導入し、次のフィールドをリストします。

```
1. Specify the following:
   a. **Favorite color**: Select your favorite color from the drop-down menu.
   b. **Favorite song**: Type your favorite song.
```

### 複数の方法

最初にわかりやすい方法を文書化します。

```
1. In the **Main Menu**, click **Home**.

   Or

   Click the **Home** icon in the upper-left corner.
```

### 直角かっこ

- メニュー/タブのナビゲーションを表示：「クリック **電子メール** > **通知**」
- 角括弧ではなく太字のボタン名
- 角括弧の周りのスペース
- メインメニューには使用しない – メインメニュースニペットを使用する

## 太字と斜体のリファレンス

### 太字 – 次の場合に使用します。

| コンテキスト | インステップ | 外部ステップ |
|---------|----------|---------------|
| クリック可能なボタン/リンク | 太字 | 大胆になるな |
| メニュー名 | 太字 | 大胆になるな |
| アイコン名 | 太字 | 大胆になるな |
| ダイアログボックス名 | 太字 | 大胆になるな |
| ページ名 | 太字 | 大胆になるな |
| フィールドリスト内のフィールド名 | 太字+コロン | 大胆になるな |

### 斜体 – 使用する目的：

- ユーザーが入力する必要があるテキスト：「URL ボックスに&#x200B;*my.workfront.com*&#x200B;と入力します」
- 強調しない – 代わりに文を再構築する

## メモ、ヒント、警告 – 詳細ガイダンス

### それぞれを使用する場合

**メモ** （`>[!NOTE]`）:
- 段落のコンテキストに合わない情報
- ユーザーの不満や時間の無駄を避ける（「このタスクを完了するにはチームライセンスが必要です」）。
- バージョン管理または互換性情報（「この機能は、従来のプルーフビューアでのみ使用できます」）

**ヒント** （`>[!TIP]`）:
- 利用者の生活を便利にするための提案は重要ではありません
- 例：「Shift キーを押しながら複数のドキュメントを同時に共有できます」

**重要** （`>[!IMPORTANT]`）:
- 手順または手順の重要な情報

**警告** （`>[!WARNING]`）:
- 潜在的なデータ損失についてユーザーに通知

### ルール

- セクションごとに1個
- 複数のメモ/ヒント/警告を絶対にスタックしない
- 関連するメモを箇条書きリストと組み合わせる
- メモを使って新しい機能を発表するのではなく、記事のテキストを変更します

## 相互参照パターン

| 状況 | パターン |
|-----------|---------|
| 情報を提供した後 | 「詳しくは、[記事の名前](link)を参照してください。」 |
| 予告なく | 「詳しくは、[記事の名前](link)を参照してください。」 |
| 別の記事のセクションへ | 「詳細については、[記事名](link)の[ セクション ]を参照してください。」 |
| 同一条中の区に | 「詳しくは、この記事の[ セクション ]を参照してください。」 |

配置：
- 段落内：インライン
- 手順の手順：手順の後の新しい行
- 複数の参照：箇条書き

## DQTI品質特性

### 使いやすい

| 特性 | 主な質問 |
|---------------|---------------|
| タスクの方向 | これは、ユーザーが何ができるのかを示すものです？ 見出しは必須動詞ですか？ |
| 精度 | 用語はUIと正確に一致していますか？ 技術的な詳細は正しいか？ |
| 完全性 | 行動するのに十分な情報はありますか？ 不要な情報を削除する必要がありますか？ |

### わかりやすいこと

| 特性 | 主な質問 |
|---------------|---------------|
| 明瞭度 | 各文はひとつの考えですか？ 言葉はシンプルで直接的なものでしょうか。 |
| 具体性 | 具体的な例はありますか？ UI要素の名前は正確ですか？ |
| スタイル | アクティブな声？ 現在時制？ 一貫した用語？ |

### 見つけやすい

| 特性 | 主な質問 |
|---------------|---------------|
| 組織 | 論理構造？ プログレッシブ・ディスクロージャー？ 最新の情報を最初に適用できますか？ |
| 取得可能性 | 読者は見出しをスキャンしてコンテンツを理解できますか？ リンクは提供されていますか？ |
| 視覚的有効性 | 3つ以上の項目のリスト？ 比較テーブル？ テキストの壁に段落を追加しない |

## 一般的な修正

| イシュー | 修正 |
|-------|-----|
| 「ご紹介できることを嬉しく思っています…」 | マーケティング言語の削除：機能を直接記述します |
| 「Xの設定が更新されました」 | 「Xを設定できるようになりました」（アクティブ、タスク指向） |
| 「それをお勧めします…」 | 「おすすめです…」 |
| 「ユーザーは機能を利用して…」 | 「この機能を使用して…」 |
| 「値を設定」/「オプションを指定」 | 「値を入力」/「オプションを選択」 |
| 「無効」/「グレー表示」 | 「利用不可」 |
| &quot;dropdown&quot; （スタンドアロン） | 「メニュー」または「ドロップダウンメニュー」 |
| 「彼/彼女」 | 「彼ら」または再構築 |
| 積み重ねノート | 箇条書き記号を使用して1つのメモに結合 |
| 太字の外部手順ステップ | 太字を削除 |
