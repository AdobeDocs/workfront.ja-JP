# 書き方

Workfront Documentation スタイルガイドとDQTIの原則を使用して、Workfront ドキュメントのレビューと改善を行います。

## ワークフロー

コンテンツを確認または編集する場合：

1. ターゲットファイルを読み込む
2. 以下のルールを適用して、問題を直接修正するか、フラグを立てます
3. 作成者の意図と技術的な正確性を確保
4. 用語、句読点、手順の詳細については、[reference.md](reference.md)を参照してください。

## ボイスと視点

- **2人目**&#x200B;を使用します（「次にできます…」） 読み手に話しかけるとき
- 会話のトーンに契約を使用する（しない、できない、できない、できない、といったように）
- 「推奨」や「必須」ではなく、「推奨」を使用してベストプラクティスを行います
- 1つのユーザーグループに対して書き込む場合は、第三者の他の役割を参照してください（「表示できます…ただし、Workfront管理者は制限できます…&quot;）
- 性別による代名詞は使わない – 文を再構築したり「they」を使ったりします
- 選択肢にスラッシュを使用しない – 「or」を使用します（「he/she」ではなく「he/she」）

## 明瞭度（DQTI）

- 1文あたりのアイデア数
- 可能な限り25文字以内の文章を残す
- 背景のコンテキストではなく、アクションまたは結果を持つリード
- 意味を伝える最も単純な単語を使用する（「使用」は「利用しない」、「開始」は「開始しない」、「about」は「about」は「regarding」ではない）
- ノミナライゼーションを避ける（「create」は「の作成」ではなく、「configure」は「の設定」ではありません）
- 具体的で具体的な言葉を使う – あいまいな用語（「いくつか」、「さまざまな」、「いくつか」）を避ける
- 二重否定を避ける
- オプションの代名詞（「that」、「who」）を使用して、文構造を明確にします

## 具体的（DQTI）

- 抽象的な説明ではなく具体的な例を用いる
- 例に現実的な値を含める
- UIの要素、フィールド、領域に正確な名前を付けます

## タスクの方向性（DQTI）

- システム *が*&#x200B;ではなく、ユーザーが&#x200B;*実行*&#x200B;できる作業に集中する
- 必須動詞を使用したリード手順（「タスクの作成」、「通知の設定」）
- ユーザーが行動するための十分なコンテキストを提供しても、それ以上は提供しない
- 詳細なヘルプへのリンクを含めます（「詳細については、[記事]を参照してください）。

## 大文字／小文字

- **見出し**：文頭の大文字と小文字を常に使用します（「タスクを作成」ではなく「タスクを作成」）
- **Workfront用語**:UI要素を直接参照する場合にのみ大文字にする
   - 直接参照：「予定完了日」フィールドを完了します。
   - 間接参照：「各タスクには完了予定日が必要です。」
   - ヒント：用語の前に「the」または「a」を配置できる場合は、通常、小文字にする必要があります
- **ボタン**：大文字と小文字を除き、UIの大文字→小文字に従う
- **ロール**: UIのロールの「システム管理者」、ユーザーの「Workfront管理者」

## 見出し

- 文頭の大文字で必須コマンドを使用する（「タスクを作成」、「アクセスレベルを設定」）
- 見出しは、タスクベースである必要があります。ユーザーが何を達成するのかを記述します
- 概要記事は「概要」で終わります（「プロジェクトの概要」）
- 明確な下位目標を用いて、長いセクションを複数の見出しに分割する

## 相互参照とリンク

以下の正確なパターンを使用します。

| 状況 | 形式 |
|-----------|--------|
| 情報を提供後のリンク | 「詳しくは、[記事の名前]を参照してください。」 |
| 事前情報のないリンク | 「詳しくは、[記事の名前]を参照してください。」 |
| 別の記事のセクションへのリンク | 「詳細については、[記事の]の[ セクション名]を参照してください。」 |
| 同じ記事のセクションへのリンク | 「詳しくは、この記事の[ セクション名]を参照してください。」 |

- 段落の場合：テキストとインライン
- 手順ステップで：手順の後の新しい行で
- 複数のリンク：箇条書きリストを使用する
- 画面の位置に「右上隅」/「左上隅」を使用する

## Bold &amp; Italics

- 手順ステップ内の&#x200B;**太字** クリック可能なアクションとUI要素のみ
- 手順ステップ以外のUI要素は太字にしないでください
- ダイアログボックス名、ページ名、アイコン名、またはメニュー名を手順の外で太字にしないでください
- ユーザーが入力する必要のあるテキストには、*斜体*&#x200B;を使用します（「URL ボックスに&#x200B;*my.workfront.com*&#x200B;と入力します」）
- 強調に斜体を使用しない – 代わりに文を再構築する

## メモ、ヒント、および警告

控えめに使用する：過剰な使用は、読者に見えなくします。

- セクションごとに最大1つのメモ/ヒント/警告
- 複数のメモを絶対にスタックしない
- 関連するメモを箇条書きリストと1つのメモに結合
- メモを使って新しい機能を知らせるのではなく、記事のテキストを変更してください

| タイプ | 目的 |
|------|---------|
| `>[!NOTE]` | 段落に合わない情報、フラストレーションを避ける、バージョン管理と互換性 |
| `>[!TIP]` | ユーザーの生活を楽にする素敵な提案 |
| `>[!IMPORTANT]` | 手順または手順の重要な情報 |
| `>[!WARNING]` | 潜在的なデータ損失についてユーザーに通知 |

## プロシージャ

- 必須のコマンド見出し（「タスクの作成」）を使用する
- 必要な場合にのみ紹介します。見出しで十分です
- 推奨されるイントロ形式：不定詞フレーズ + コロン（「一時的なパスワードを作成するには：」）
- 1つの手順では、番号付きリストを使用します
- 「type」または「select」を使用 – 「set」や「specify」などの曖昧な単語を使用しない
- 条件付き手順：「（条件付き）複数のチームのメンバーである場合は、選択してください…」
- オプションの手順：「（オプション）絵文字を追加するには、をクリックします…」
- システム応答を事前または直後に記述する
- 複数の方法：最初に最も簡単な方法を文書化し、「または」を使用します

### プロシージャの直角かっこ

- `>`を使用してメニュー/タブのナビゲーションを表示します。「クリック **電子メール** > **通知**」
- 角括弧ではなくボタン名を太字にする
- 角括弧の周りにスペースを含める
- メインメニューナビゲーションには角括弧を使用しない – メインメニュースニペットを使用する

## 句読点のクイックリファレンス

| ルール | 例 |
|------|---------|
| オックスフォード・コンマは常に | 「動機、手段、および機会」 |
| 手順の「then」の前にコンマを挿入 | 「設定をクリックしてから、「インターフェイス」をクリックします。」 |
| 年末年始の日付の中文 | “ The January 2, 2016, issue of ...” |
| 月のみのコンマはありません | 「2016年1月」 |
| リストの前にコロン、後に小文字 | 「次のいずれかを選択：オプション A」 |
| プロシージャ見出しの後にコロンがありません | 「タスクの作成」（「タスクの作成」ではありません） |
| エムはダッシュを控えめに | 可能であれば文を作り直す |
| ファイル拡張子の期間 | 「pdf ファイルのアップロード」 |

句読点と用語の詳細については、[reference.md](reference.md)を参照してください。

## 完全性（DQTI）

- 利用者が把握して行動するために必要なあらゆる情報を含める
- 利用者が必要としない情報を含めないでください
- 詳細ドキュメントへの「詳細」リンクを必ず提供してください
- ドキュメントシステムのデフォルト：リストの「（デフォルト）」、または文の説明

## 組織（DQTI）

- プログレッシブ開示付き論理構造（概要→詳細→参照）
- 時間順序付きページ（リリースノートなど）では最新のコンテンツが最初に表示される
- セクションごとに1つのトピック
- 3つ以上の並列項目にリストを使用
- 構造化された比較またはフィールドの説明にテーブルを使用する
- テキストの壁で構成される段落を避け、理解しやすい塊に分ける

## 記事の種類

| タイプ | 目的 | タイトル条約 |
|------|---------|-----------------|
| 概要 | コンテクストと図法を組み合わせて活用 | 「概要」で終わる |
| ハウツー | 明確なステップによる具体的なタスク | 必須動詞 |
| 基本を学ぶ | 設定情報+新規ユーザー用のリンク | 「基本を学ぶ…」 |
| リファレンス | テーブルのフィールド説明 | 記述名詞 |

## 検証チェックリスト

編集後、次の項目を確認します。

- [ ]全体で2人目（「あなた」）、契約書使用済み
- [ すべての見出しに]個の文頭のみ大文字
- [ ]個のWorkfront用語が正しく大文字に変換されました（直接参照と間接参照）
- [ プロシージャ ステップ内のクリック可能なアクションに対してのみ]太字
- [ ]相互参照は標準形式を使用しています（「詳細については、を参照してください…」）
- [ ]件のメモ/ヒント/警告が積み重ねられていません。1つのセクションにつき1件まで
- [ ] Oxford コンマが一貫して使用されています
- [ ]手順の手順では、特定の動詞を使用します（「set」または「specify」ではなく、タイプ、選択、クリック）
- [ ]条件付き/オプションの手順が正しくラベル付けされています
- [ ]性別の代名詞がありません
- [ ]個の文章が明確、具体的、タスク指向です
