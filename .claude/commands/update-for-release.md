# リリース用アップデート（Workfront）

このスキルでは、今後の機能リリースに向けてWorkfrontのヘルプ記事を更新する方法について説明します。 ワークフローは`remove-preview-highlighting`の逆です。新しい動作が記事に追加され、プレビューとしてマークされ、その他のスキルによって（後でGAで）クリーンアップされます。

## 範囲

**all**&#x200B;がtrueの場合に適用：

1. ユーザーは、出荷中の機能（通常は最初にプレビュー）のWorkfront ヘルプ記事を更新しています。
2. この変更により、GA クリーンアップではなく、新しい動作またはUIが導入されます。 GA クリーンアップの場合は、代わりに&#x200B;**remove-preview-highlighting**&#x200B;を使用してください。
3. ファイルはリリースノートの&#x200B;**not**&#x200B;です。 リリースノートの場合は、代わりに&#x200B;**release-notes-formatter**&#x200B;を使用してください。
4. 少なくとも簡単な説明とスクリーンショットを提供します。理想的にはPRD URL （Adobe Wiki）です。

範囲が明確でない場合は、開始する前に確認してください。

## 必要なワークフロー（人間が介在）

リポジトリを&#x200B;**not**&#x200B;一括編集します。 記事を1つずつ移動します。 各記事の後、次に進むかどうかを尋ねます。

### &#x200B;1. 機能コンテキストの収集

ユーザーに確認します。

- **変更されたもの** （新しい動作またはUIの1～2文の要約）。
- 新しいUIの&#x200B;**スクリーンショット**。 指定した場合は、説明的なケバブケースのファイル名を含むターゲット記事の`assets/` フォルダーに保存します（例：`add-custom-message.png`）。 指定しない場合は、1つを待つか、プレースホルダー参照で続行するかを尋ねます。
- **PRD URL** （Adobe Wiki）を使用します（使用可能な場合）。 `user-Adobe Wiki Confluence` MCP ツール `get_wiki_content`で取得します。 UIでユーザーが見られない動作を見つけるために読んでください。通知の副作用、後で何かが編集または追加されたときに何が起こるか、文字制限が表示されないか、権限など。
- **可用性**: プレビューのみ、プレビュー+高速リリース、または既に一般公開されています。 これにより、ステップ 3でのスニペットの選択が促進されます。
- **明示的な除外**：ユーザーがスキップしたい記事（例：「この機能はテンプレートにありません」）。

### &#x200B;2. 影響を受ける記事のインベントリ

機能領域のキーワード（例：`approval workflow`、`document approval`、特定のフィールドラベル）を使用してリポジトリを検索します。 候補リストを作成する：

- 関連する`help/quicksilver/.../` ツリーのハウツー記事。
- 機能領域について言及した概要記事とFAQ。
- **除外** `product-announcements/` （リリースノートでは異なるスキルが使用されています）。
- **別の記事へのテキストのリンクのみが記載されている**&#x200B;目次/索引ページを除外します。
- **手順1でスキップするとユーザーが言った**&#x200B;件の記事を除外します。

候補リストをユーザーに提示します。 どの更新を行い、どの更新をスキップするかを指定します。 兄弟の記事が見つからない場合は、`help/quicksilver/TOC.md`を相互参照します。

### &#x200B;3. プレビュースニペットの選択

`help/_includes/snippets.md`を読み取り、利用可能条件で選択します：

| 使用可否 | スニペット |
| --- | --- |
| プレビューのみ – ハイライト表示されたコンテンツは、一般公開の記事では新しく追加されています | `{{highlighted-preview}}` |
| プレビューのみ – 記事全体が新しい | `{{highlighted-preview-article-level}}` |
| プレビュー+高速リリースのお客様、一般 | `{{preview-fast-release-general}}` |

現在の四半期にリリース固有のスニペットが既に存在する場合は、汎用的なものよりも優先します。 お申し込みの前にユーザーに選択を確認してください。

### &#x200B;4. 記事ごとに – 最初に表示し、OKの後に編集

ユーザー確認リストの各記事について：

1. **ファイルを読み込みます。**

2. **強調表示パターンを決定します。** この記事に当てはまるものをユーザーに尋ねます（回答は記事ごとに異なる場合があります）。

   - **セクションごとの複製**：既存のセクション見出しに`in Production`を追加します。 `in Preview`が追加され、`<div class="preview"> ... </div>`で囲まれた新しいセクションを追加します。 新しいビヘイビアーによって手順が意味を持って変更される場合（追加の手順、新しい画像、新しい表の行、異なる表現など）に使用します。 典型的なチュートリアル手順です。
   - **行単位の回り込み**：既存のセクション内に新しい文をインラインで追加し、`<span class="preview"> ... </span>`で折り返します。 追加が既存の段落、表のセル、またはFAQの回答に自然に収まる文または2である場合に使用します。
   - **混在**：同じ記事の一部のセクションでセクションごとの複製が使用され、その他のセクションでは行ごとのラッピングが使用されます。 記事にプロシージャルセクションとFAQ スタイルのセクションの両方がある場合は、このオプションを表示します。

3. **H1見出しの直後に、スニペット**&#x200B;を独自の行に配置し、上下に空白行を配置します。 スニペットは、イントロ段落の&#x200B;**前**&#x200B;に配置されます。

4. **新しい詳細を「常に含める」と「レビュー中」にバケット化します。** これは最も重要なステップです。

   - **常に**&#x200B;を含める（自動的に適用、プロンプトなし）:UIの操作中にユーザーが確認できない非表示の動作。 例：
      - 副作用（例：「これを編集すると、すべての参加者に電子メールが再送信される」）
      - 他のオブジェクトまたは後のイベントでの動作
      - 前提条件と権限
      - UIに表示されない制限
      - ユーザーがPRD、ドキュメント、製品チームからしか学べないもの
   - **レビュー用にアップ** （`AskQuestion`を複数選択したユーザーに表示）：機能の使用中にユーザーが画面に表示できる事実。 例：
      - UIに既に表示されている文字カウンター（例：`0 / 500`）
      - フィールドのデフォルトの拡張/折りたたみ状態
      - 表示されるチェックボックスの既定の選択状態
      - フィールドの横にあるボタンラベル
      - インラインに表示される検証メッセージ

   「レビュー中」の項目ごとに、1文の根拠を入力します（「初心者がより長いメッセージを計画するのに役立つ」、「後の段階でメッセージを見ないユーザーが拡大することを知るのに役立つ」）。 ユーザーが選択した項目のみを含めます。 デフォルトの原則は、「タスクを行っているときに画面に表示できる場合は、それを再表示しないでください」ですが、ユーザーは最終的な呼び出しを受け取ります。

5. **編集を提案します。** スニペットの配置、見出しの変更、新しいプレビュー内コンテンツ、その位置、スクリーンショット参照、インライン `class="preview"`回の回り込みなど、記事の前/後の抜粋（またはフォーカスされた差分スタイルの説明）を表示します。

6. **ファイルを書き込む前に、明示的な承認** （「okay」、「apply」、「yes」）を待ちます。

7. **検証します。** 書き込み後、ファイルで`ReadLints`を実行し、問題を報告します。 構造を確認するには、変更されたセクションをもう一度読みます。

8. **文章レベルの編集**&#x200B;を&#x200B;**書き込み品質** スキルに延期します。 音声、大文字、太字のルール、またはリンク パターンをここでやり直さないでください。解析パスが必要な場合は、`~/.cursor/skills/writing-quality/SKILL.md`をお読みください。

### &#x200B;5. 各記事の後

次の記事に移動するか、現在の記事を停止するか、スキップするか、再訪問するかを尋ねます。

### &#x200B;6. セッションの終了 – リリースノートのコピー/ペースト

ユーザーがセッションを終了したとき（「完了」と表示された場合、「完了」、「停止」、または次の記事に進むのを拒否した場合）、次の質問を行います。

> 「機能強化ページのリリースノートのコピー/ペーストを行いますか？」

はいの場合は、手順1で更新した機能コンテキストと、このセッションで更新されたプライマリヘルプ記事を使用してドラフトエントリを生成します。 **どのファイルにも書き込まないでください** — テキストのコピー/ペーストのみとして指定します。

**release-notes-formatter** スキルの製品領域ページ構造に合わせてエントリを書式設定します。

```markdown
## {Feature name}

>[!NOTE]
>
>Preview: {date or TBD}
>Production fast release: {date or TBD}
>Production for everyone: {date or TBD}

{1–3 sentences describing what changed and why it helps users. Lead with the benefit, not the UI action.}

For more information, see [{Primary article title}](/help/quicksilver/{path-to-article}.md).
```

ルール：

- 日付が不明な場合は`TBD`を使用します。ユーザーに日付があるかどうかを確認します。
- 機能名は文頭の大文字です（先頭の単語と固有名詞のみを大文字にします）。
- 説明では、実装の詳細ではなく、ユーザーが今できることに焦点を当てる必要があります。
- 概要ページではなく、更新された最も具体的なハウツー記事へのリンク。
- すべての日付が不明で、ユーザーがプレースホルダーを望まない場合は、`>[!NOTE]`日付ブロックを含めないでください。これを省略して、後で追加する必要があることに注意してください。

## コンテンツルール

### 見出し

- 実稼動側の参照として残る既存のセクション見出しに正確に&#x200B;**`in Production`**&#x200B;を追加します。
- 新しいセクション見出しに正確に&#x200B;**`in Preview`**&#x200B;を追加します。
- 見出しの残りの部分は文頭の大文字にしてください（`writing-quality`）。

### ラッパーのプレビュー

- **セクションレベル**: `<div class="preview"> ... </div>`で終了します。 各タグの上と下に空白行を配置し、その中のマークダウン見出しとリストをレンダリングします。
- **インライン （文レベル）**：既存の段落、表のセル、またはFAQの回答の中で`<span class="preview"> ... </span>`に折り返します。
- `<span class="preview">`を`<div class="preview">`内にネストしない。

### スニペット配置

- スニペット行はH1の直後に行われ、上下に空白行が表示されます。
- スニペットは、イントロ段落、`>[!IMPORTANT]`の吹き出し、およびアクセス要件ブロックの&#x200B;**前**&#x200B;に配置されます。
- 記事ごとに1つのスニペット。

### Screenshots

- 記述的なケバブケースのファイル名を使用して、記事の`assets/` フォルダーに新しいスクリーンショットを保存します。
- 新しいプレビュー内セクション内から新しいスクリーンショットを参照します。 実稼動中のセクションのスクリーンショットが機能を正確に反映しなくなった場合は、その機能を残しておきます。これは、GAまで実稼動動作を表します。
- スクリーンショットのファイル名を作成しないでください。まだスクリーンショットが提供されていない場合は、ユーザーに問い合わせてください。

### メモとヒント

- セクションごとに最大1つの`>[!NOTE]` （または`>[!TIP]`、`>[!IMPORTANT]`、`>[!WARNING]`）です。 既存のセクションに既にメモがある場合は、関連する新しいコンテンツを積み重ねるのではなく、箇条書きリストと同じメモに組み合わせます。

### してはいけないこと

- `product-announcements/`の下の記事は編集しないでください。
- 一括編集しないでください。一度に1つの記事を作成し、毎回明示的に承認します。
- 最初にユーザーに表示せずに、識別可能なUIの事実を含めないでください。
- ユーザーが明示的に指示しない限り、`<!-- ... -->`個のHTML コメント内のコンテンツを変更しないでください。
- `author:`または無関係なフロントマターフィールドは変更しないでください。

## 編集内容を提示する前に品質チェックを行う

- スニペットは、H1の後に、上下に空白行を含む独自の行に1回表示されます。
- 既存のセクション見出しは`in Production`で終わります。
- 新しいセクション見出しは`in Preview`で終わり、セクションは`<div class="preview">`内にあります。
- インラインの追加は`<span class="preview">`内にあります。
- `ReadLints`は編集したファイルでクリーンです。
- 記事は両方の状態で正しく読み取られます（プレビューコンテンツが表示され、非表示になります）。
