# [Data Connect]の[Job History]ビューの使用

「Job History」ビューでは、Workfront管理者は各データ更新ジョブの詳細なレコードにアクセスできます。 これらの記録は、データを最新の状態に保つジョブに関する貴重な情報を提供し、プロセスを実行してビジネスの可視化を更新するタイミングを理想的な時間枠に設定するのに役立ちます。

![ジョブ履歴ビュー](assets/job-history-overview.png)

[Job History]ビューの列には、次の情報が表示されます。

* **OBJECT_NAME**:ジョブに関連付けられているオブジェクトの名前を表示します。
* **SCHEDULED_TIME**:ジョブの開始時刻を表示します。
* **COMPLETED_TIME**:ジョブの完了時刻を表示します。
* **LATEST_FLAG**:ジョブが最新の更新の一部であったかどうかを示します。
* **状態**:ジョブの状態を表示します。 詳細については、この記事の次のセクションを参照してください： [利用可能なジョブステータス](#available-job-statuses)。
* **LAST_UPDATED**:ジョブの最終更新タイムスタンプです。

>[!NOTE]
>
>[Job History]ビューには、スケジュールされた過去72時間のジョブの詳細が表示されます。


## 使用可能なジョブ・ステータス

各データ接続ジョブには、成功、スキップ、または失敗したかどうかを示すステータスが割り当てられます。

<table>
    <tr>
        <td><b>ジョブステータス</b></td>
        <td><b>定義</b></td>
    </tr>
    <tr>
        <td>成功</td>
        <td>ジョブは使用可能な各更新を正常に処理し、そのレコードタイプのすべての更新がデータレイクに反映されるようになりました。</td>
    </tr>
    <tr>
        <td>スキップ</td>
        <td>レコードの種類に対して処理されるキューの更新がないため、ジョブはスキップされました。</td>
    </tr>
    <tr>
        <td>失敗</td>
        <td>ジョブを実行できませんでした。 このような場合、キュー内のデータがデータレイクにコミットされていない可能性があります。 キューに残っているレコードは、そのレコードタイプの次にスケジュールされたジョブで処理されます。 </td>
    </tr>
   </table>


## ジョブの実行とロギングの動作に関する考慮事項

Snowflakeでは、ジョブの実行方法と[ジョブ履歴]ビューでのログ方法に影響するジョブスケジューラ最適化ツールを使用します。 このログの動作は、処理するデータがあるかどうかによって異なる可能性があります。

例えば、特定のオブジェクトに対して処理する新しい行がない場合、次のいずれかの結果が生じる可能性があります。

* **Jobが実行され、Skippedとしてマークされています**: Snowflakeは、処理する行がないことを検出し、ジョブを実行して、テーブルにSkippedステータスで記録します。

* **ジョブは実行されません**: Snowflakeは、処理する行がないことを確認し、ジョブを実行せず、テーブルにスキップされたステータスでジョブを記録します。

  >[!NOTE]
  >
  >ジョブが実行されない2つ目のシナリオでは、そのオブジェクトの最新のレコードに、予定されたスケジュールに一致しないタイムスタンプが反映されている場合があります。

つまり、ローが処理されていなくてもジョブは実行されたと見なされ、ジョブスケジューラの特定のジョブに対する動作に応じてログに記録されるかどうかが決まります。
