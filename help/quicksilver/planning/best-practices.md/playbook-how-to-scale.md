# プレイブック：最初の勝利の後、スケーリングを管理

**バージョン**:1.0

**コンテキスト**:「最初のユースケースを見つけましたが、今ではすべての人が望んでいます」



## 1.「サクセストラップ」

WFP 導入の最も危険な段階は、最初の成功したユースケースまたは POC の直後であると感じる場合があります。 熱意は大きいが、「技術的負債」と「行政の無秩序な拡大」に対する恐れは、次の 2 つの同じように悪影響を及ぼす反応につながりかねない。

1. **オーバーガバナンス**：システムを非常に厳重にロックすることで、新しいチームはスプレッドシートに戻ります。

2. **ゼロガバナンス**：各チームが独自のフィールドとレコードタイプを作成し、従来の環境に見られる断片化されたメタデータを再現します。



## 2.中心理念「和解エンジン」としての WFP

チームの違いを止めようとするのではなく、WFP を、それらの違いが **見えるようにして）調整できる** 場所と位置づけています。



* **導入速度の管理**：既存の環境を「クリーンアップ」する前に、新しいツールに拡張することに慎重になるのは自然です。 **シンプル化を優先** を選択すると、高度に管理された基盤が提供されますが、連携の準備が整ったチームの価値創出を遅らせる可能性があります。 この変化を先導する最も効果的な方法は、**可視性はステップ 1** であることを認識することだと考えています。 共有されたエンタープライズ対応ツール（PPT やスプレッドシートの増加から脱却する）への勢いが、最終的に長期目標の妨げとなっています。



  **提言**:「最初にクリーンアップする」権限ではなく、継続的なメンテナンスにはより少ないリソースを、差し迫ったビジネスニーズの解決にはるかに多くのリソースを割り当てることをお勧めします。 例えば、1 年を「一掃」する分類だけに費やすと、増分値はほとんどありません。 ただし、統合エンタープライズカレンダーや統合 GTM ロードマップなどを通じて **チーム間の表示** を実現することで、関係者が必要とする変革の価値が提供されると同時に、環境を経時的に管理するために必要な統合データ構造が提供されます。

* **現実を認識**：独立したチームは、当然のことながら独自のプロセスを開発しますが、多くの場合、サイロ化されたスプレッドシートに隠されたままになります。 WFP への移行は混乱を生まず、既に存在するものだけに光を当てます。 これらのプロセスを共有環境で表示することにより、最終的に対処して改善できる場所にプロセスを配置できます。

* **進捗シグナル**：チームが独自のフィールドを要求する場合、それは「無秩序」ではなく、**導入** です。 つまり、WFP をワークスペースとして認識しています。

* **負債の管理、非表示にしない**：後で異なる分類をクリーンアップするために必要な労力について懸念するのは当然です。 しかし、厳格な基準を早く導入しすぎるという代替策により、チームはスプレッドシートに戻され、そこでプロセス（および負債）が隠されたままになっていることがよくあります。 チームが現在の分類から WFP で開始できるようにすることで、その負債を目に見える管理された環境に移行します。 これにより、最終的な調整は、単一の圧倒的な移行プロジェクトではなく、反復的なタスクになります。



## &#x200B;3. 「道路上の車線」ガバナンスモデル

「グローバルレーン」と「ローカルプレイグラウンド」を定義することで、負債なしで拡張できます。



### A. グローバル・レーン

* **制御オブジェクト**：すべてのチームがエンタープライズレポートに使用する必要があるオブジェクト（例：`Strategic Pillar`、`Region`、`Fiscal Quarter`）。

* **管理者**：一元化された COE/マーケティング業務管理者。

* **ルール**：これらのフィールドは「共有」されており、必須です。



### B. ローカル・プレイグラウンド（スポーク）

* **実験的オブジェクト**：チームの戦術的ニーズに固有のフィールドまたはレコードタイプ（ソーシャルチームの `Influencer Handle` や web チームの `URL Slug` など）。

* **管理者**: チームリード（ライトガイダンス付き）。

* **ルール**：チームはここでイノベーションを行うことができます。 「ローカル」フィールドを 3 チーム以上が採用した場合は、グローバルレーンに「昇格」されます。



## &#x200B;4. ガバナンスのパラドックス：まずチーム、基準に従う

WFP を拡大する際の一般的な課題は、**エンタープライズガバナンス** または **チームの運用上の連携** のどちらが優先されるかを決定することです。



この問題を解決する最も効果的な方法は、企業価値が双方向の道で構築されていることを認識することです。

1. **チームには関連性が必要**：企業は、チームがアクティブに実行されている場合にのみ価値を実現します。 したがって、ガバナンスは、既知の運用ニーズをサポートする構造を提供することで **チームにサービスを提供** する必要があります。

2. **企業には可視性が必要**：データが集約できるほどクリーンな場合にのみ、リーダーシップは十分な情報に基づいた意思決定を行うことができます。 したがって、チームは、ポートフォリオの表示に必要な最小限の実行可能なメタデータを提供することで **企業に貢献** する必要があります。



「マネージドスケーリング」の目標は、これら 2 つのニーズの積集合を見つけることです。これは、可視性を提供するために十分な標準化ですが、チームの実行を遅らせることではありません。



### A.優先度の調整：データとビジュアライゼーションの比較

拡大縮小する場合は、「値」の定義がペルソナ間で異なることを認識します。

* **管理者/製品所有者**：値 **統合された分類と分類**。 彼らの目標は、長期的なスケーラビリティをサポートするクリーンなデータアーキテクチャです。

* **関係者/リーダー**：値 **ビジュアライゼーションとinsight** （グローバルカレンダー、Portfolio タイムラインなど）。 目標は、データを実用的にする「稲妻の瞬間」です。



**戦略**：関係者のビジュアライゼーションのニーズを、チームが管理者のデータ標準規格のニーズに準拠するための *インセンティブ* として使用します。 リーダーシップが必要とする「スーパーカレンダー」を提供することで、統一された分類を得ることができます。



### B. 「サービス主導」の観測段階

初期のスケールアップ段階では、管理者の役割は、チームと企業の間でのこの交換を容易にすることです。

* **標準化よりも運用の優先順位を付ける**：グローバルな定義の欠如によって立ち往生するよりも、サイロ化されたワークスペースでチームが積極的に計画を立てる方が良いでしょう。 このアクティビティは、健全な実際の標準を構築するために必要な「生データ」です。

* **「表示最小限」の特定**：リーダーシップと連携して、エンタープライズレポートに明確にする *必要* 3～5 個のフィールド（`Strategic Alignment`、`Start Date`、`Budget` など）を特定します。 ここで強制エネルギーを集中してください。



### ロ。遡及調和（サービスとしてのガバナンス）

チーム間で「既知のニーズ」のパターンが発生したら、企業は、それらのパターンをグローバルサービスに統合するために移行できます。

1. **成功するパターンを観察**：チームが既に構築し、採用した「勝者」分類を特定します。

2. **共同ハンドシェイク**：チームチャンピオンを集めて、地元での成功を共有された企業標準に磨き上げます。

3. **サービスとしてのデプロイ**：新しいグローバルレーンを「制限」としてではなく、作業を行うユーザー向けのレポートとチーム間の連携を簡素化する方法として展開します。



**重要な留意点**：ガバナンスは、運用の成功に対する対応であるべきで、そのための前提条件であってはならない。



## &#x200B;5. スケーリングの仕組み：パターンに基づく成長モデル

この哲学を適用するには、データ構造に対する思慮深いアプローチが必要です。 「ガバナンスの乱立」を避けるために、個々のリクエストごとにグローバルなフィールドを構築したいという衝動に抵抗します。 代わりに、**フィールド成熟度パス** を使用して、実際の使用状況にエンタープライズ標準を導くことができます。



1. **レベル 1：ローカル実験**：チーム A がワークスペースにカスタムフィールドを作成します。

2. **レベル 2：パターン認識**：管理者は、チーム B と C が類似のフィールドを使用または要求していることを確認します。

3. **レベル 3：企業の標準化**：管理者は、**グローバル分類Workspace** のレコードタイプとして、そのフィールドの標準化された単一バージョンを作成し、チームに同時配信します。



### 「ソフトリタイアメント」の仕組み

WFP には現在、フィールドに対するネイティブの「アーカイブ」機能がないので、ローカルフィールドを廃止する場合、UI を混乱させずに履歴データを保持するために、意図的な「ソフト廃止」プロセスが必要です。



1. **データ移行**：テーブルビュー（または Fusion）を使用して、ローカルの「シャドウ」フィールドから新しいグローバルレーンフィールドに値を一括コピーします。 この移動中に、データが検証され、消去されていることを確認します。

2. **非推奨にする名前を変更**:`[DEPRECATED]` や `z_` などのプレフィックスを使用してローカルフィールドの名前を変更します（例：`z_Language (Old)`）。 これにより、フィールドがフィールドピッカーの下部にプッシュされ、「真実のSource」ではなくなったことをユーザーに知らせます。

3. **フォームの削除**: **これは最も重要な手順です。** すべての **レコードのForms** から非推奨フィールドを削除します。 これにより、必要に応じて既存のテーブルビューやレポートに古いデータを表示したまま、新しいデータが入力されるのを防ぐことができます。

4. **「サンセット」期間**：移行中にデータが失われないように、非推奨フィールド（プレフィックスとオフ形式）を 30～60 日間保持します。 この期間を過ぎると、グローバルレーン内でデータが完全に紐付けされた場合、ローカルフィールドをワークスペースから削除できます。



## 6.「コアドリフト」（抽象化規則）の回避

Planning がコアのように散乱するのを防ぐには、次の手順に従います：

* **抽象化レイヤー**:WFP のすべてのフィールドは、**戦略的な質問** に回答する必要があります。 フィールドが戦術的なトラッキングにのみ使用される場合（例：「このプルーフは承認されましたか」）、コアで保持します。

* **最初に統合**：チームが新しいメタデータフィールドを必要としている場合は、まずグローバル分類を確認するようにチームを招待します。 これには、チームのリード **読み取り専用アクセス** をグローバル分類ワークスペースに付与する必要があります（セクション 7 を参照）。 戦術的なニーズを既存の戦略的フィールドにマッピングすることで、不要な重複を防ぎ、レポートの整合性を維持できます。



## &#x200B;7. 「読み取り専用アクセス」表示モデル

「サイロ化」された仕事の騒音なしで「サイロ化」された感覚を解決します。

* **問題**：スポークのチームは、自分のレコードしか表示されないので孤立していると感じます。

* **問題を修正**：共有レコードタイプに対して、**プライマリ」として指定されたワークスペースへの読み取り専用アクセス** をチームに付与します。

* **結果**：インスピレーションと連携のためにより広いエンタープライズコンテキストを確認できますが、ローカルワークスペースはクリーンなままで、特定のタスクに焦点を当てています。



## 8.ワークショップを通じた成長管理

WFP の拡大は、技術的な課題と同じくらい文化的な課題です。 「ガバナンスのギャップ」を埋めるために、的を絞ったワークショップを実施する。



### A. 「必要な混乱」発見ワークショップ

* **対象読者**：地域のマーケティングリードや Ops チャンピオン。

* **目標**：現在の「サイロ化された現実」（断片化された運用データ）を文書化します。

* **メッセージ**:「フィールドを削除する権限がありません。 それらがグローバル戦略にどのように結び付いているかを理解するために、私たちはここにいます」

* **成果**：地域の戦術分野をグローバルな戦略的レーンにマッピングする草案。



### B. 「戦略的な可視性」の整合性セッション

* **対象読者**：マーケティングのハイレベルな関係者（リーダーシップ）。

* **目標**:「最初からシンプルにする」不安の枠組みを変えます。

* **メッセージ**:「開始するのに、完璧な分類は必要ありません。 私たちは、WFP を環境として使用して *完璧な分類を構築*」しています。

* **結果**：コアが現在の状態のままの間に、調整エンジンとして WFP を使用して前進するための承認。



### ハ。「スポーク・ツー・グローバル」ショーケース

* **対象読者**:WFP を調査する新しいチーム。

* **目標**:「サイロ化」された感じを軽減します。

* **メッセージ**:「チーム A のローカル作業が、指定されたプライマリWorkspaceをどのように自動的にフィードするかを確認します。 自分の作業に対しても、同じ可視性を持つことができます。」

* **結果**：地域の独立性を失うことなく「接続」されるメリットを享受できる新しい部門をオプトインします。



### D. 「継続的サポート」営業時間

* **対象読者**：すべての WFP ユーザー（現在および将来のユーザー）。

* **目標**：トラブルシューティングや戦術的なガイダンスのために、低コストの繰り返し環境を提供します。

* **メッセージ**:「間違った質問はありません。 お客様が特定の計画上の課題をリアルタイムで解決できるよう、アドビがサポートします。」

* **結果**：ユーザーの信頼の向上、技術的摩擦の迅速な解決、国際的な標準化が必要となる可能性のある新しいパターンの特定。



## 9.規模に応じた人材の確保：役割・責任

マネージドスケーリングモデルの成功には、単なるツール設定だけでなく、グローバルチームとスポークチームに明確な役割を分散させる必要があります。



### A. エンタープライズアーキテクト（中央 COE/マーケティング業務）

* **焦点**：企業の整合性、システムパフォーマンスおよび **統合データ分類**

* **責務**:

   * **グローバル分類Workspace** を管理します。

   * **フィールド成熟度パス** を容易にします（グローバル標準に対するローカルの成功を促進します）。

   * エグゼクティブレポート用の **プライマリWorkspace** ビューを管理します。

   * ワークスペース全体で毎月の **セマンティック監査** をリードします。



### B. スポーク・チャンピオン（チーム・プロセス・オーナー）

* **焦点**：チームの関連性と導入の速度。

* **責務**:

   * 部門チームの単一の連絡窓口として機能します。

   * ローカルワークスペース構造とカスタムフィールド実験を管理します。

   * チームがデータ入力に **管理されたゲートウェイ** （Forms）を使用していることを確認します。

   * ハーモナイゼーション中に **協調ハンドシェイク** に参加します。



### C. エグゼクティブスポンサー（マーケティングリーダーシップ）

* **フォーカス**：戦略的な連携、OKR の可視性および **ポートフォリオビジュアライゼーション（グローバルカレンダーなど）**。

* **責務**:

   * グローバル分類ワークスペースのエンタープライズ **マーケティング OKR** を定義します。

   * [ 可視性ステップ 1] の値を他の引出線に対して支持します。

   * 80/20 リソース配分（クリーンアップに対する値）を強制的に指定します。



### D. イネーブルメントリード（変更管理）

* **フォーカス**：文化のシフトとスキルの開発。

* **責務**:

   * **Office Hours** および **ディスカバリーワークショップ** を定期的に開催します。

   * 内部の「成功事例」ショーケースを維持します。

   * 解決するエンタープライズアーキテクトの技術的な摩擦ポイントを特定します。



## 10.次回チーム拡大チェックリスト

* [ ]&#x200B;**チャンピオンの特定**：この新しいチームの「プロセスオーナー」または「チャンピオン」とは誰ですか？

* [ ] **「ローカルデルタ」の定義**：このチームでは、グローバル標準で現在提供されていない 2 ～ 3 個のフィールドのうち、必要なものはどれですか？

* [ ]&#x200B;**グローバルレーンへのマッピング**：既存のグローバルフィールドのうち、ニーズの 80% を満たすことができるのはどれですか？

* [ ]&#x200B;**グローバルな表示の許可**:1 日目に、関連するプライマリワークスペースとグローバルな分類ワークスペースへの読み取り専用アクセス権を付与します。

* [ ]&#x200B;**ハンドオフの確立**：彼らの作業は、関連するプライマリワークスペースにどのように「フィード」されますか？ （例：グローバルレコードタイプまたは特定の参照を使用）。
