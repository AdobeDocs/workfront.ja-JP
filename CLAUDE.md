# 新しい記事を作成する前に候補記事を検索する

新しいコンテンツを組み込む場合（受け入れドキュメント、お客様またはPM フィードバック、Slack/文字起こしペースト、メモ付きスクリーンショット、サポートに関する質問）、**最初に新しい記事を作成することを提案しません。** リポジトリを検索し、コンテンツが収まる可能性のある既存の記事の候補を表示します。

## 必要なワークフロー

1. **新しいコンテンツを最初に読み**&#x200B;して、トピック、オーディエンス（管理者とエンドユーザーの比較）、追加される具体的な事実や手順を特定します。
2. **grepを使用して`help/`で候補**&#x200B;を検索し、見つけます。 同じ機能、分野、ワークフローに対応している記事を探しましょう。
3. **次の各項目を含む候補記事のランクリスト**&#x200B;を返します（通常は3～7）。
   - ファイルパス（クリック可能なバックティックパス）。
   - 候補である1行の理由（どのセクションにスロットインするか、またはなぜそれが最も近いトピックの一致である理由）。
   - 注意事項（例：「オーディエンスの不一致 – これはエンドユーザーであり、取り込みは管理者向けです」）
4. **起草または編集の前に、コートニーに依頼**&#x200B;します。 質問をわかりやすく表現します。例えば、「どの質問に草稿を書くべきか」などです。 あるいは、「#2に合わせたいのか、それとも独自の記事が必要だと思うのか？」と聞いてきます。
5. **既存の記事が適切でない場合にのみ新しい記事を提案**&#x200B;し、候補者が誰も機能しない理由を説明します。

「明らかに」コンテンツが既知の記事に属している場合でも、コートニーが配置を確認できるように、2-3の代替案が表示されます。 これにより、ほぼ重複するデータや、コンテンツが既に部分的にカバーされている場所を特定できます。

# ヘルプ記事を編集する前に変更点を提案

`help/`の下の`.md` ヘルプ記事を編集しようとすると、**まだファイルを編集しません**。 まず、変更する予定のものを提示し、明示的な承認を待ちます。

タッチするファイルごとに以下を行います。

1. ファイルに名前を付けます（パス）。
2. 提案された変更を、フェンス付きの差分/スニペット（古いテキストと新しいテキスト）として表示します。周囲のコンテキストは明確にします。
3. 理由を簡潔に述べる（1文）。
4. 明確な質問で終わる（例：「これらの変更を適用しますか？」） または「続行しますか？」

コートニーが「はい」（または「行く」、「適用」、「実行」など）と答えた後にのみ表示されます。 エディターツールを呼び出す必要があります。

これは、ヘルプ `.md`記事の&#x200B;**すべての**&#x200B;編集に適用されます。タイプミス、リンクの修正、単語入れ替え、用語のクリーンアップ、新しいセクション、書き換えです。 コートニーが同じターンで「ただそれを修正する」または「要求しない、単に編集する」と明示的に言わない限り、例外はありません。

複数の記事にまたがる変更の場合：1回の返信で提案された差分をファイルごとにグループ化します。 セットが大きい場合は、最初に影響を受けるファイルを1行の要約でリスト化し、次にバッチで差分を表示し、各バッチの前に確認します。

これは、読み取り専用の調査（grep、読み取り）やチャットでのドラフト/探索（ファイルの書き込みなし）を&#x200B;**not** ブロックしません。

# Git コミットメッセージ

Git コミットメッセージを作成または生成する場合は、次の形式に従います。

## 件名

- **50文字以下**&#x200B;程度です。
- **必須ムード**&#x200B;の変更を要約します（例：「追加…」、「修正…」、「リファクタリング…」）。

## 本文

- 本文の前の被写体の後に1行の空白行を残します。
- 行を約&#x200B;**72文字**&#x200B;で折り返します。
- 説明には&#x200B;**箇条書き**&#x200B;を使用します。 各箇条書きは、次のいずれかから始める必要があります。
   - **📖** – 新しいファイル、新しいセクション、新しい機能、新しいヘッダー、新しい行、その他のグリーンフィールドコンテンツなど、変更&#x200B;**によって新しい項目が**&#x200B;追加された場合に使用します。
   - **✏️** – 変更&#x200B;**が**&#x200B;の既存の作業を変更する場合に使用します。既存の行の編集、既存のセクションの更新、リファクタリング、または現在のコンテンツの変更です。

例：

```
Add refresh token rotation to auth flow

- 📖 Add refresh_tokens table and Alembic migration for schema v3.
- ✏️ Update session middleware to rotate secrets and revoke old tokens.
```

# プルリクエスト（PR）

## Jiraの問題の導出

- **現在のGit ブランチ名**&#x200B;からJira課題IDを派生させます（例：`FFENT-8796`に一致するセグメント）。
- **ブランチ名にJira IDが含まれる場合：** PR タイトルにそのIDを使用し、`# Context`→`## Jira`にリンクします。
- **ブランチ名にJira IDが含まれていない場合：** Jira キーをPR タイトルから除外します。 `## Jira`を正確に含めます：`No ticket`。

## PR タイトル

**Jira ID:** `{type}/{JIRA-ID}- {short task description}`
例：`feat/FFENT-8001- Add validation for numVariations in OCAPI request`

**Jira IDなし：** `{type}- {short task description}`
例：`docs- Refresh Object Composite API changelog`

### コミットタイプ

- **feat** – 新機能を追加します
- **fix** — バグの修正
- **refactor** – 動作を変更せずにコードを書き換えたり再構築したりします
- **perf** — パフォーマンスの向上
- **style** – 書式/空白のみ。意味は変更されません
- **test** — テストの追加または修正
- **docs** — ドキュメントのみ、新規コンテンツが追加されました
- **build** — ビルドツール、CI、依存関係、プロジェクトバージョン
- **ops** — インフラストラクチャ、デプロイメント、バックアップ、リカバリ
- **chore** – その他（例：`.gitignore`）

## PR body — required sections

### `# Summary`
何が変化し、なぜ変化したのかを簡潔に説明します。

### `# Changes`
ファイルで整理。 タッチされた各ファイルに対して、バックティックのパスにレベル 2の見出し、次に箇条書き記号を使用します。

```markdown
# Changes

## `path/to/file.ext`
* Concise bullet describing the change in that file.
```

### `# Context`
常に`## Jira` サブセクションを含めます。

```markdown
# Context

## Jira
[FFENT-8796](https://jira.corp.adobe.com/browse/FFENT-8796)
```

チケットがない場合：`No ticket`
